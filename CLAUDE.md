## Code Generation

- Always TypeScript strict mode
- Prefer `type` over `interface` for data shapes unless inheritance or augmentation is explicitly needed
- Use **named exports** — no default exports except Vue single-file components (once Vue is confirmed)
- Write small, composable functions — no god objects or god files
- Game logic must be **framework-agnostic** — pure TypeScript modules, zero UI framework imports
- Use the exact terminology from `AI_DIRECTIVES.md` in all type names, variable names, and filenames
- Centralize domain constants — no magic numbers or magic strings inline

### Suggested Module Boundaries (early guidance, not locked)
```
src/
  app/                        # Vue 3 SPA — all frontend code
    engine/                   # Pure TS game engine — graph, flow simulation, card registry
    stores/                   # Pinia stores — useUserStore, useGameStore, useBoardStore, useSimStore
    components/               # Vue components
    composables/              # Shared Vue composables

  api/                        # Hono Worker — all backend code
    routes/                   # Route handlers — save, device, catalog, version
    middleware/               # Auth (Hanko), logging, CORS

  packages/                   # Shared between app and api — each has its own package.json
    schemas/                  # Zod schemas — API payloads, save JSON content, auth data
    db/                       # Drizzle schema, drizzle-zod, DB client
      migrations/             # SQL files generated by drizzle-kit — never hand-written
    types/                    # Shared TS types not derivable from any schema — keep very thin
    constants/                # Named constants shared across app and api
```

#### Monorepo rules
- Root `package.json` declares workspaces pointing to `src/app`, `src/api`, `src/packages/*`
- `app` and `api` import shared code as internal workspace packages — not via relative `../../` paths
- Engine lives in `src/app/` — it is client-side only; engine types that need to cross the app/api boundary get promoted to `packages/types`
- Do not add code to `packages/` that is only used by one side — it belongs in `app/` or `api/` directly

### Store Rules
- **Setup style only** — never use Pinia options style (`state/getters/actions` object)
- **One store per concern** — do not add responsibilities to an existing store because it's convenient; flag it and discuss
- **`useBoardStore`** — use `shallowRef` for card and connection collections; trigger reactivity manually
- **`useSimStore`** — flat `ref<number>` per value; no deep reactive objects
- **Do not** put UI state (panel open/closed, selected card, hover state) in any of the four stores — that belongs in component-local state or a dedicated `useUIStore` if it needs to be shared
- Stores are the bridge between engine and UI — they do not contain game logic
- User preferences (theme, routing style, sound) belong in `useUserStore` — not component local, not a separate UI store

### Zod
- Use at data boundaries only — IndexedDB reads, API responses, KV data, auth data
- Do not use Zod for internal engine state or store-to-component data flow
- Always use `z.infer<>` to derive TS types — never write manual types for anything with a schema
- Save schemas use `z.discriminatedUnion('version', [...])` for migrations — no if/else version branching
- `src/schemas/` holds non-DB schemas only — API payloads, save state JSON content, auth data
- DB model schemas come from drizzle-zod and live in `src/db/` — never duplicated in `src/schemas/`

### Database (Drizzle)
- `src/db/schema.ts` is the single source of truth for all table definitions
- Never write migrations by hand — always `drizzle-kit generate` then commit the output to `src/db/migrations/`
- Never write TypeScript types or Zod schemas for DB models manually — derive from Drizzle schema via drizzle-zod
- `docs/migrations/` is for notes about why a migration happened — not SQL files
- The JSON `data` blob in the saves table is validated by a separate Zod schema in `src/schemas/save.ts`, not by Drizzle

## Architecture Guidance

- The **graph engine** (Cards, Connections, Flow) is a standalone pure-TS module — build it first, independently of any UI
- The **UI layer** consumes graph state reactively — it does not own or mutate game logic directly
- **Cycle detection** must be built into the graph from the start — do not defer it as an edge case
- When a Connection is removed, the engine handles flow interruption and pooling — UI reflects it, does not drive it
- Connection routing is a **UI concern**, not an engine concern — the engine tracks source/target ports only; the UI layer handles path geometry and routing style
- When a Card moves, the UI must re-route all attached Connections automatically
- Design all card archetypes as extensions of a base Card contract — no special-casing archetypes in the engine core
- Cycle detection is an engine responsibility — must be in place before any Cycle-capable Card type is implemented

### Save & Sync Layer
- **IndexedDB is the source of truth** — never write game state anywhere else first
- The sync layer is a separate module; it must not be entangled with the game engine or UI
- **Auth state drives sync availability** — the sync module checks Hanko session state before any cloud operation
- **Offline is the default assumption** — all API calls must be wrapped in try/catch with silent failure; the UI receives a connection status signal, not raw errors
- **Device ID** is generated once per browser, stored in IndexedDB, never changes for that browser instance
- **Conflicts only occur when two tabs are simultaneously active and saves have diverged** — normal handoff (close tab → open on other device) is always seamless
- **Save versioning:** every save has `version` (increments on every write) and `lastSyncVersion` (version at last successful sync); these are the source of truth for conflict detection — not timestamps, not presence alone
- **Timestamps stay in saves** for display in Save Snapshot UI only — do not use for conflict logic (clock skew)
- **Incremental/delta saves** — deferred; flag as future optimization if save payload size becomes a concern
- **Device presence = live heartbeat in KV** — short TTL (~2-3 min); `beforeunload` marks inactive as best-effort; presence is a UX signal, not the conflict trigger
- **Conflict detection is API-side** — compares `version` ancestry against `lastSyncVersion`; returns `SAVE_CONFLICT` with both snapshots if diverged; client never detects conflicts itself
- **Kill Flag is short-lived** — ~5-10 min TTL; losing tab is open right now and will pick it up on its next heartbeat
- The conflict modal is a UI concern — sync module emits `saveConflict` event; UI handles display and returns resolution choice
- Never auto-resolve a Save Conflict; never write a Kill Flag without a user decision
- `SAVE_CONFLICT` is a typed API response code — define it as a constant, not a magic string

## Ask Before Deciding

Always ask the user before making a call on:
- Any framework or library not yet confirmed
- Any new currency type or card archetype
- Persistence / save-game approach
- Any major structural pattern (ECS, event bus, pub/sub, reactive streams, etc.)
- Anything that would affect the module boundary suggestions above

Proceed without asking on:
- TypeScript type definitions for confirmed domain concepts
- Internal file/folder structure within a confirmed module
- Naming of private helpers that don't affect public API contracts
- Refactoring or cleanup within already-reviewed code
